# 2025-02-17: プロダクト方向性の議論

## 議論の流れ

### 1. 目的と手段の整理（ドリルと穴の比喩）
- ContextGitは「ドリル」。ユーザーが本当に欲しい「穴」は何かを議論
- 4つの「穴」を特定:
  - A: AI記憶の統合（各AIに散らばったコンテキストの一元管理）
  - B: 再説明の排除（AIが自分を知っている状態）
  - C: 最適な答え（タスクに応じた最適AI自動選択）
  - D: AIナレッジ管理（成功プロンプト・ワークフローの蓄積と再利用）
- 当初Cの優先度を低く見積もったが、ユーザーからの修正でCも重要と確認。A〜D全てが本質

### 2. オーケストレーション統合レベルの整理
- レベル1: ContextGit + Claude in Chromeは外部ツール（統合なし）
- レベル2: ContextGitが外部AI結果を取り込める（小規模拡張）
- レベル3: ContextGit自体がオーケストレーション能力を持つ（作り直し級）
- レベル4: ContextGitがメタAIプラットフォーム（完全別物）
- **結論: レベル3+4のハイブリッド**。デフォルトは自動（L4）、ユーザーが指定もできる（L3）

### 3. 動的DAGワークフローの具体化
- パワポ作成の例: Claude→Gemini/ChatGPT並列→Claude→ChatGPTレビュー
- 調査レポートの例: 3トピック×3AI=9並列タスク→統合→再検索→成果物
- ContextGitは「AIワークフローエンジン」であり、タスクDAGを動的に構築・実行する

### 4. GitモデルとAIワークフローの対応
- Branch/Worktree → 並列AI実行
- Commit → 各AI出力のスナップショット
- Merge → 複数AI結果の統合
- PR + Review → 統合結果のレビュー（別AIがレビューア）
- Fork → 別バージョン・別アプローチの分岐
- Main → 最終成果物
- 本質: ソフトウェア開発で解決済みの問題をAI活用に転用

### 5. ユーザー層とプロダクト構造
- 非エンジニアとエンジニア（自分含む）の2層
- **1プロダクト2モード**に決定
- エンジニア向けから先に作り、非エンジニア向けUIを後から被せる
- 参考: Git→GitHubの関係

### 6. 技術手段の検討
- CDP MCP（Google提供）: フルブラウザ制御、Claude非依存だが一般ユーザーには困難
- Claude in Chrome: 既に動作するが Claude依存
- Chrome拡張API: ユーザーフレンドリーだが能力に制約
- **結論: 未決定**。一般ユーザー向けはChrome拡張API、エンジニア向けはCDP MCPが有力

### 7. プロダクト優先方針
- ハッカソン等のイベント出展よりプロダクト開発を優先
- 中途半端になるリスクを避ける

### 8. ドキュメント管理方針
- LLMはgit履歴を見ないという課題
- `contextgit-vision.md` = LLMが読む唯一のファイル（現在のビジョン + 決定ログ）
- `discussions/` = 時系列の詳細記録（アーカイブ）

## 主要な決定事項
1. ContextGitの本質は「AIチームのためのGitHub」
2. レベル3+4ハイブリッドオーケストレーション
3. 1プロダクト2モード（エンジニア先行、非エンジニアUI後載せ）
4. プロダクト開発優先（イベント出展より）
5. ドキュメントは vision.md（LLM向け） + discussions/（アーカイブ）の2層構造
